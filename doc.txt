1. PROJECT OVERVIEW
1.1 Deskripsi
FinTrack adalah sistem akuntansi keuangan personal berbasis double-entry bookkeeping, dibangun dengan arsitektur cloud-first menggunakan Vercel (serverless) + Supabase (PostgreSQL) + Telegram Bot sebagai primary input interface. Sistem dirancang scalable — COA, workflow, dan schema sudah production-ready sehingga bisa dijadikan template UMKM atau PT tanpa rebuild.
1.2 Tujuan Sistem
Catat transaksi keuangan personal secara benar (double-entry, audit trail)
Input harian via Telegram Bot tanpa buka laptop
Dashboard web untuk monitoring, laporan, dan analisis
Scalable ke bisnis: COA, workflow, schema siap untuk UMKM/PT
Transaksi salah = REVERSE (bukan delete) — data tidak pernah hilang

1.4 Prinsip Akuntansi
PRINSIP
IMPLEMENTASI DI FINTRACK
Double-Entry Bookkeeping
SUM(Debit)=SUM(Kredit) wajib sebelum setiap POST — enforced di app layer
Period Lock
/lock YYYY-MM → periode dikunci, tidak bisa input/edit transaksi di bulan itu
Audit Trail
Semua INSERT/UPDATE di transactions & journal_lines → auto-log ke audit_log via trigger
No Delete Policy
Transaksi salah: buat dokumen REVERSE (RV) di tanggal hari ini — data asli tetap ada
COA 3-Level
Kode:Akun:Sub-Akun — scalable dari personal ke PT, 74 akun total
Accrual-Ready / Cash Default
Arsitektur mendukung accrual, default perilaku cash basis untuk personal
Imprest Method
SeaBank = kas kecil, target Rp 500.000, isi otomatis hitung selisih ke target
Normal Balance
Aset & Beban = Debit normal. Liabilitas, Ekuitas, Pendapatan = Kredit normal

3.1 Database Views
View: account_balances
Query saldo realtime per akun. Dipanggil oleh /saldo di bot dan BalanceCard di dashboard.
CREATE VIEW account_balances AS
SELECT coa.code, coa.account_name, coa.account_type, coa.normal_balance,
COALESCE(SUM(jl.debit_amount),  0) AS total_debit,
COALESCE(SUM(jl.credit_amount), 0) AS total_credit,
CASE WHEN coa.normal_balance = 'debit'
THEN COALESCE(SUM(jl.debit_amount),0) - COALESCE(SUM(jl.credit_amount),0)
ELSE COALESCE(SUM(jl.credit_amount),0) - COALESCE(SUM(jl.debit_amount),0)
END AS balance
FROM chart_of_accounts coa
LEFT JOIN journal_lines jl ON coa.code = jl.account_code
LEFT JOIN transactions t   ON jl.doc_number = t.doc_number AND t.status = 'POSTED'
WHERE coa.is_header = false AND coa.is_active = true
GROUP BY coa.code, coa.account_name, coa.account_type, coa.normal_balance;
View: monthly_summary
Agregasi income vs expense per bulan. Dipanggil /bulan di bot dan Reports page di dashboard.
CREATE VIEW monthly_summary AS
SELECT t.period_year, t.period_month, coa.account_type,
COALESCE(SUM(jl.debit_amount),  0) AS total_debit,
COALESCE(SUM(jl.credit_amount), 0) AS total_credit
FROM transactions t
JOIN journal_lines jl ON t.doc_number = jl.doc_number
JOIN chart_of_accounts coa ON jl.account_code = coa.code
WHERE t.status = 'POSTED' AND coa.is_header = false
GROUP BY t.period_year, t.period_month, coa.account_type;

7.1 CRITICAL — Harus Disolve Sebelum Deploy
#
BLINDSPOT
RISIKO
SOLUSI
B1
Supabase koneksi port 5432 (direct) bisa exhausted di serverless
Pool habis, DB tidak bisa diakses
Pakai port 6543 (PgBouncer) + ?pgbouncer=true di connection string di shared/db.py
B2
Telegram webhook perlu didaftarkan manual setelah deploy ke Vercel
Bot tidak menerima pesan sama sekali
Jalankan scripts/setup_webhook.py setelah setiap deploy: POST api.telegram.org/botTOKEN/setWebhook?url=...
B3
React SPA routing: refresh /journal → 404 di Vercel tanpa catch-all route
User tidak bisa bookmark atau refresh halaman
Di vercel.json: routes [-1] = {"src":"/(.*)", "dest":"/index.html"} — HARUS paling akhir
B4
bot_state race condition: 2 Telegram updates bersamaan bisa corrupt state
Bot double-post atau state kacau
Wrap read+write bot_state dalam Supabase transaction. Gunakan SELECT FOR UPDATE atau upsert atomic.
B5
Auth cookie httpOnly tidak bisa dibaca JavaScript di React
ProtectedRoute tidak bisa cek apakah user logged in
Wajib ada /api/auth/me endpoint yang return {logged_in: true/false} — di-call saat mount
B6
/setup wizard bisa double-run → Opening Balance di-post dua kali
Saldo awal salah (double)
Sebelum run /setup: cek existence OB document. Jika ada → tampil info OB sudah ada + saldo saat ini.
B7
Vercel Python function size limit 250MB — python-telegram-bot cukup besar
Deploy gagal atau cold start lambat
Test deploy dulu tanpa semua deps. Kalau terlalu besar: ganti ke pyTelegramBotAPI atau raw httpx.

9. CHANGELOG — v2 & v3 (dokumen ini di atas cuma cerita v1)
9.1 Kenapa ada bagian ini
Dokumen di atas (bagian 1-8 + appendix COA) adalah build plan v1 ASLI dan tidak pernah
diupdate sejak — v2 dan v3 dikerjakan lewat sesi terpisah dan didokumentasikan di file
lain (docs/CHANGELOG_v2.md, docs/API.md, README.md). Bagian ini cuma RINGKASAN supaya
doc.txt tetap jadi satu titik referensi; untuk detail penuh baca file aslinya.
9.2 v2 — OCR Struk + API Layer Integrasi (2026-07-03)
Ringkasan lengkap: docs/CHANGELOG_v2.md. Poin utama:
- Foto/screenshot struk → OCR (OCR.space free tier) → parse (merchant/nominal/tanggal)
  → konfirmasi via bot → posting KK. Tabel baru: receipts.
- API layer integrasi: POST /api/transactions (buat transaksi dari luar) + POST
  /api/receipts/parse, auth cookie ATAU header X-API-Key (FINTRACK_API_KEY, opsional).
- Keputusan arsitektur: proposal migrasi ke FastAPI+Railway DITOLAK — tetap 100% Vercel
  serverless + Supabase free, demi same-origin auth & nol biaya baru.
9.3 Migrasi Frontend ke TypeScript + Tailwind + React Query + Zod + Vitest/Playwright (2026-07-08)
Dashboard React (src/) yang tadinya JS polos + CSS tangan + cache fetch buatan sendiri,
di-rewrite total: TypeScript di semua src/, Tailwind v4 (tema glassmorphic/gradient,
lihat src/components/AppBackground.tsx), komponen ala shadcn/ui (Radix + class-variance-
authority, src/components/ui/*), Framer Motion (transisi halaman), React Query
(gantikan cache manual di hooks/), Zod (validasi form + validasi runtime bentuk
response API), plus test suite baru: Vitest (unit/komponen) + Playwright (e2e, API
di-mock karena login asli lewat magic-link Telegram tak bisa dites headless).
Blindspot yang dibenahi di migrasi ini: error API yang tadinya didiamkan sekarang
ditampilkan, sesi yang expired di tengah pemakaian sekarang redirect ke /login
otomatis, tidak ada lagi layar putih kalau ada render error (ErrorBoundary), label
form di Settings sekarang terhubung ke input-nya (aksesibilitas), logout butuh
konfirmasi, field angka di-validasi Zod (bukan string mentah), dan format Rupiah
untuk nominal negatif diperbaiki.
9.4 v3 — Budget/Goals/Recurring/Bills/Tags/Forecast/Laporan Terjadwal (2026-07-09)
Batch fitur terbesar sejauh ini — ~40 item dari review kelayakan "harus tetap 100%
gratis" (lihat artifact review yang dibagikan terpisah). Semua yang lolos review
(gratis atau tinggal tweak) diimplementasikan; yang butuh biaya (fitur LLM) atau
scope besar (split expenses jadi app "keuangan bersama") sengaja DI-SKIP.

Tabel & kolom baru (db/12-17_*.sql — belum dijalankan ke Supabase produksi, harus
dijalankan manual sama seperti db/11_receipts.sql dulu):
- budgets, goals, recurring_transactions, bills, tags, transaction_tags, activity_log
- chart_of_accounts.is_custom (tandai kategori beban yang ditambah user sendiri)
- Preferensi baru di bot_settings: currency_preference, timezone, daily/weekly_report_
  enabled, alert_sensitivity, rate_limit_per_minute, budget_alert_throttle_mins

Modul Python baru (shared/): forecast.py (z-score + regresi linear, stdlib murni,
tanpa numpy — konsisten prinsip B7 anti-bloat), fx.py (konversi mata uang via
frankfurter.app, gratis tanpa API key), retry.py, sparkline.py, activity.py
(backing store rate limiting + anti-abuse + activity log sekaligus), scheduling.py.

Bot (api/telegram/webhook.py) — command & wizard baru: /undo (reverse transaksi
terakhir 1 tap), /budget /budgets, /goal /goals, /recurring [list], /bill /bills,
/tag /tags, /kategori add, /convert. Plus: rate limiting per user, pesan error umum
yang lebih ramah (detail teknis di-log ke server bukan dikirim ke user), peringatan
anti-abuse (nominal tak wajar) di preview sebelum posting, alert budget real-time
setelah posting pengeluaran (throttled), DAN — perbaikan alur atas permintaan user:
setiap posting sukses (pengeluaran/pemasukan/transfer) sekarang menampilkan tombol
"lanjut transaksi" di bawah pesan sukses, jadi tidak perlu ketik /menu atau /start
lagi untuk input berikutnya (lihat continue_keyboard()).

API baru/diperluas: /api/budgets (+?resource=goal), /api/recurring (+?resource=bill),
/api/tags (+?action=assign), POST /api/accounts (kategori custom), /api/reports
(satu function, dibedakan ?report=balance|monthly|ledger|trial-balance|
income-statement|range|forecast), savings_rate di ?report=monthly,
filter ?tag= di GET /api/transactions.

⚠️ BLINDSPOT KETEMU SAAT DEPLOY PERTAMA (sudah diperbaiki): batch v3 awalnya
nambah 13 file .py baru di atas 12 file lama = 25 Serverless Functions total.
Vercel Hobby plan cuma boleh maksimal 12 Serverless Functions per deployment —
deployment pertama GAGAL (Vercel diam-diam fallback ke build lama, jadi semua
endpoint baru 404 padahal kelihatannya "sudah di-push"). Fix: 7 file reports/*
digabung jadi api/reports/index.py (dispatch via query param ?report=), begitu
juga budgets+goals -> api/budgets/index.py (?resource=), recurring+bills ->
api/recurring/index.py (?resource=), tags+assign -> api/tags/index.py
(?action=assign), dan 5 file cron -> api/cron/index.py (?job=). Total balik ke
persis 12 function. Pelajaran: di proyek ini, setiap tambah endpoint API baru
HARUS dicek dulu total jumlah file .py di api/ (termasuk subfolder) tidak lewat
12 — kalau mepet, gabung ke function yang sudah ada lewat query-param dispatch
alih-alih bikin file baru.

Cron (api/cron/index.py, satu function dibedakan ?job=daily|weekly|monthly|
annual|housekeeping, dipicu .github/workflows/scheduled-reports.yml) — auth
pakai CRON_SECRET terpisah dari FINTRACK_API_KEY (beda tujuan, beda
blast-radius kalau bocor):
- daily (09:00 WIB): ringkasan hari ini + auto-post recurring transactions jatuh
  tempo + reminder tagihan H-3
- weekly (Minggu): breakdown kategori + sparkline + deteksi anomali (z-score,
  keputusan user: mingguan bukan bulanan) + progress goals
- monthly (tgl 1): perbandingan bulan lalu vs bulan sebelumnya
- annual (1 Jan): ringkasan tahunan + top 5 kategori beban
- housekeeping (tiap 30 menit): sapu bot_state yang macet >30 menit, notify user
  proaktif (sebelumnya cuma reset pasif tanpa pemberitahuan)

Dashboard (src/): 4 halaman baru (Budgets, Goals, Recurring, Bills) dengan CRUD
lengkap — sesuai keputusan user, dikelola dari BOT dan DASHBOARD sekaligus, bukan
salah satu saja. Tambahan di halaman lama: kategori custom di COA, kelola tags +
preferensi v3 di Settings, filter tag di Jurnal, export CSV/JSON/PDF (selain Excel
yang sudah ada), tab rentang-tanggal-bebas + forecast di Laporan, savings-rate di
Dashboard.

Testing: 60 test Python baru (pytest — forecast, activity/rate-limit, fx, scheduling,
simulasi wizard bot lewat pemanggilan fungsi langsung karena tidak ada token bot/
Supabase asli di lingkungan dev), tambahan test Vitest (skema Zod, util CSV,
komponen) dan Playwright e2e (4 halaman baru, API di-mock).

9.7 Review keamanan (cybersecurity pass) (2026-07-09)
Bukan simulasi fungsional — review khusus permukaan keamanan: auth/session,
RLS/hak akses DB, CORS, penanganan secrets, CVE dependency, permukaan XSS,
SSRF/DoS di endpoint OCR. Ditemukan 4 hal, 2 sudah diperbaiki, 2 didiskusikan
dan sengaja DIBIARKAN atas keputusan user.

Diperbaiki:
1. Race condition (TOCTOU) di magic-link login (api/auth/verify.py): cek
   is_used=false dan tandai is_used=true sebelumnya 2 langkah terpisah
   (SELECT lalu UPDATE) — dua request nyaris bersamaan dengan token yang sama
   (single-use, dari /getlink) bisa lolos SELECT sebelum salah satunya sempat
   UPDATE, menghasilkan 2 sesi dari 1 token yang seharusnya cuma bisa dipakai
   sekali. Fix: digabung jadi SATU UPDATE ... WHERE is_used=false — Postgres
   row-lock otomatis cuma meloloskan satu request, request kedua tidak match
   baris apa pun. Eksploitasi butuh penyerang sudah pegang token itu duluan
   (HMAC-signed, sekali pakai, kadaluarsa dalam hitungan menit) jadi risiko
   praktis rendah, tapi tetap bug TOCTOU nyata dan gampang diperbaiki.
2. api/receipts/parse.py (endpoint OCR struk) menerima body image_base64 dan
   men-decode SELURUH payload sebelum ada pengecekan ukuran — satu-satunya
   guard ukuran (ocr.MAX_BYTES) ada di shared/ocr.py, SETELAH decode. Endpoint
   ini sudah butuh auth (cookie sesi atau X-API-Key) jadi bukan celah publik,
   tapi tetap ditambah penolakan dini kalau base64-nya jauh melebihi batas —
   jaga-jaga dari pemborosan resource/kuota OCR.space gratis.
3. CRON_SECRET didokumentasikan di .env.example (sebelumnya sudah dipakai
   nyata oleh api/cron/* sejak v3, tapi tidak pernah ditulis di file contoh
   env — celah dokumentasi, bukan celah keamanan).

Ditemukan tapi SENGAJA dibiarkan (keputusan user, bukan lupa):
4. Webhook Telegram (api/telegram/webhook.py do_POST) fail-open kalau
   WEBHOOK_SECRET belum di-set — cek signature dilewati dan endpoint menerima
   POST dari siapa saja, termasuk berpotensi spoof from.id supaya lolos
   whitelist OWNER_TELEGRAM_ID. User mengonfirmasi WEBHOOK_SECRET sudah
   di-set di Vercel, jadi ini bukan celah aktif — dibiarkan fail-open (bukan
   diubah ke fail-closed) karena WEBHOOK_SECRET memang env var opsional by
   design ("aman saat rollout" awal), dan ide hardening ke fail-closed
   sengaja tidak dieksekusi supaya tidak berisiko mengunci bot sendiri kalau
   env var itu suatu saat hilang tanpa sengaja.
5. Package xlsx@0.18.5 (dipakai untuk export .xlsx di src/utils/exportXlsx.ts)
   punya 2 CVE publik (prototype pollution, ReDoS) yang cuma dipatch di
   versi 0.19.3+ — versi itu sudah tidak dipublish ke npm registry (SheetJS
   pindah distribusi patch ke CDN resminya sendiri). Dicek: app ini CUMA
   pernah panggil XLSX.utils.json_to_sheet/book_new/writeFile (tulis), TIDAK
   PERNAH XLSX.read/readFile (baca) — jalur kode yang rentan (parsing file
   asing) tidak pernah dieksekusi sama sekali di app ini. User memilih tetap
   di versi npm biasa (bukan pindah ke sumber dependency non-npm CDN SheetJS)
   karena risiko nyata sudah mendekati nol dan instalasi npm standar lebih
   diutamakan; catatan ini disimpan supaya diingat kalau nanti ada fitur
   IMPORT spreadsheet — saat itu risikonya jadi relevan dan perlu dicek ulang.

Tidak jadi masalah (dicek, aman): db/08_security.sql sudah RLS ON di semua
