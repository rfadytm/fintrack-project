1. PROJECT OVERVIEW
1.1 Deskripsi
FinTrack adalah sistem akuntansi keuangan personal berbasis double-entry bookkeeping, dibangun dengan arsitektur cloud-first menggunakan Vercel (serverless) + Supabase (PostgreSQL) + Telegram Bot sebagai primary input interface. Sistem dirancang scalable — COA, workflow, dan schema sudah production-ready sehingga bisa dijadikan template UMKM atau PT tanpa rebuild.
1.2 Tujuan Sistem
Catat transaksi keuangan personal secara benar (double-entry, audit trail)
Input harian via Telegram Bot tanpa buka laptop
Dashboard web untuk monitoring, laporan, dan analisis
Scalable ke bisnis: COA, workflow, schema siap untuk UMKM/PT
Transaksi salah = REVERSE (bukan delete) — data tidak pernah hilang

1.4 Prinsip Akuntansi
PRINSIP
IMPLEMENTASI DI FINTRACK
Double-Entry Bookkeeping
SUM(Debit)=SUM(Kredit) wajib sebelum setiap POST — enforced di app layer
Period Lock
/lock YYYY-MM → periode dikunci, tidak bisa input/edit transaksi di bulan itu
Audit Trail
Semua INSERT/UPDATE di transactions & journal_lines → auto-log ke audit_log via trigger
No Delete Policy
Transaksi salah: buat dokumen REVERSE (RV) di tanggal hari ini — data asli tetap ada
COA 3-Level
Kode:Akun:Sub-Akun — scalable dari personal ke PT, 74 akun total
Accrual-Ready / Cash Default
Arsitektur mendukung accrual, default perilaku cash basis untuk personal
Imprest Method
SeaBank = kas kecil, target Rp 500.000, isi otomatis hitung selisih ke target
Normal Balance
Aset & Beban = Debit normal. Liabilitas, Ekuitas, Pendapatan = Kredit normal

3.1 Database Views
View: account_balances
Query saldo realtime per akun. Dipanggil oleh /saldo di bot dan BalanceCard di dashboard.
CREATE VIEW account_balances AS
SELECT coa.code, coa.account_name, coa.account_type, coa.normal_balance,
COALESCE(SUM(jl.debit_amount),  0) AS total_debit,
COALESCE(SUM(jl.credit_amount), 0) AS total_credit,
CASE WHEN coa.normal_balance = 'debit'
THEN COALESCE(SUM(jl.debit_amount),0) - COALESCE(SUM(jl.credit_amount),0)
ELSE COALESCE(SUM(jl.credit_amount),0) - COALESCE(SUM(jl.debit_amount),0)
END AS balance
FROM chart_of_accounts coa
LEFT JOIN journal_lines jl ON coa.code = jl.account_code
LEFT JOIN transactions t   ON jl.doc_number = t.doc_number AND t.status = 'POSTED'
WHERE coa.is_header = false AND coa.is_active = true
GROUP BY coa.code, coa.account_name, coa.account_type, coa.normal_balance;
View: monthly_summary
Agregasi income vs expense per bulan. Dipanggil /bulan di bot dan Reports page di dashboard.
CREATE VIEW monthly_summary AS
SELECT t.period_year, t.period_month, coa.account_type,
COALESCE(SUM(jl.debit_amount),  0) AS total_debit,
COALESCE(SUM(jl.credit_amount), 0) AS total_credit
FROM transactions t
JOIN journal_lines jl ON t.doc_number = jl.doc_number
JOIN chart_of_accounts coa ON jl.account_code = coa.code
WHERE t.status = 'POSTED' AND coa.is_header = false
GROUP BY t.period_year, t.period_month, coa.account_type;

⚠️ BLINDSPOT KETEMU SAAT DEPLOY PERTAMA (sudah diperbaiki): batch v3 awalnya
nambah 13 file .py baru di atas 12 file lama = 25 Serverless Functions total.
Vercel Hobby plan cuma boleh maksimal 12 Serverless Functions per deployment —
deployment pertama GAGAL (Vercel diam-diam fallback ke build lama, jadi semua
endpoint baru 404 padahal kelihatannya "sudah di-push"). Fix: 7 file reports/*
digabung jadi api/reports/index.py (dispatch via query param ?report=), begitu
juga budgets+goals -> api/budgets/index.py (?resource=), recurring+bills ->
api/recurring/index.py (?resource=), tags+assign -> api/tags/index.py
(?action=assign), dan 5 file cron -> api/cron/index.py (?job=). Total balik ke
persis 12 function. Pelajaran: di proyek ini, setiap tambah endpoint API baru
HARUS dicek dulu total jumlah file .py di api/ (termasuk subfolder) tidak lewat
12 — kalau mepet, gabung ke function yang sudah ada lewat query-param dispatch
alih-alih bikin file baru.

Cron (api/cron/index.py, satu function dibedakan ?job=daily|weekly|monthly|
annual|housekeeping, dipicu .github/workflows/scheduled-reports.yml) — auth
pakai CRON_SECRET terpisah dari FINTRACK_API_KEY (beda tujuan, beda
blast-radius kalau bocor):
- daily (09:00 WIB): ringkasan hari ini + auto-post recurring transactions jatuh
  tempo + reminder tagihan H-3
- weekly (Minggu): breakdown kategori + sparkline + deteksi anomali (z-score,
  keputusan user: mingguan bukan bulanan) + progress goals
- monthly (tgl 1): perbandingan bulan lalu vs bulan sebelumnya
- annual (1 Jan): ringkasan tahunan + top 5 kategori beban
- housekeeping (tiap 30 menit): sapu bot_state yang macet >30 menit, notify user
  proaktif (sebelumnya cuma reset pasif tanpa pemberitahuan)

9.7 Review keamanan (cybersecurity pass) (2026-07-09)
Bukan simulasi fungsional — review khusus permukaan keamanan: auth/session,
RLS/hak akses DB, CORS, penanganan secrets, CVE dependency, permukaan XSS,
SSRF/DoS di endpoint OCR. Ditemukan 4 hal, 2 sudah diperbaiki, 2 didiskusikan
dan sengaja DIBIARKAN atas keputusan user.

Diperbaiki:
1. Race condition (TOCTOU) di magic-link login (api/auth/verify.py): cek
   is_used=false dan tandai is_used=true sebelumnya 2 langkah terpisah
   (SELECT lalu UPDATE) — dua request nyaris bersamaan dengan token yang sama
   (single-use, dari /getlink) bisa lolos SELECT sebelum salah satunya sempat
   UPDATE, menghasilkan 2 sesi dari 1 token yang seharusnya cuma bisa dipakai
   sekali. Fix: digabung jadi SATU UPDATE ... WHERE is_used=false — Postgres
   row-lock otomatis cuma meloloskan satu request, request kedua tidak match
   baris apa pun. Eksploitasi butuh penyerang sudah pegang token itu duluan
   (HMAC-signed, sekali pakai, kadaluarsa dalam hitungan menit) jadi risiko
   praktis rendah, tapi tetap bug TOCTOU nyata dan gampang diperbaiki.
2. api/receipts/parse.py (endpoint OCR struk) menerima body image_base64 dan
   men-decode SELURUH payload sebelum ada pengecekan ukuran — satu-satunya
   guard ukuran (ocr.MAX_BYTES) ada di shared/ocr.py, SETELAH decode. Endpoint
   ini sudah butuh auth (cookie sesi atau X-API-Key) jadi bukan celah publik,
   tapi tetap ditambah penolakan dini kalau base64-nya jauh melebihi batas —
   jaga-jaga dari pemborosan resource/kuota OCR.space gratis.

   select() list manual dan kelewatan satu kolom (endpoint lain pakai
   select("*") atau RPC dengan kontrak kolom tetap, jauh lebih aman dari jenis
   bug ini).
2. Export Laba Rugi ke Excel tidak ada baris Pendapatan: exportIS() di
   src/pages/Reports.tsx dump baris per-akun revenue/expense tanpa baris
   subtotal — kalau bulan itu tidak ada akun pendapatan dengan transaksi

   baru dapat bobot lebih besar — teknik forecasting deret waktu pendek yang

   membengkokkan tren jangka panjang. Winsorizing sengaja pakai z-score
   "leave-one-out" (rata-rata & stdev dihitung dari titik-titik LAIN, bukan
   termasuk titik yang diuji) — z-score biasa gagal total di sini karena
   classic "masking effect": satu outlier ekstrem menggelembungkan stdev-nya
   sendiri sampai z-score-nya mentok di sekitar sqrt(n-1) tidak peduli
   seberapa besar nilainya, jadi tidak pernah lewat threshold berapa pun.
   Hasil forecast di-floor ke 0 (tidak pernah negatif).

tests/test_forecast.py +5 (redam outlier tunggal, tidak pernah negatif,
_winsorize menahan titik normal tapi meredam outlier, regression guard
khusus untuk masking effect). Total pytest 80 → 86. tsc bersih.

9.14 Forecast bertingkat (pendek/menengah/panjang) + landasan bulan lengkap (2026-07-09)
Tindak lanjut §9.13's perbaikan forecast: user tanya "kapan nilai forecast
bisa keluar" dengan konteks baru aktif 9 hari. Ditelusuri: window forecast
lama SELALU mundur `months` (default 6) bulan kalender dari HARI INI, tidak
peduli kapan user mulai pakai sistem — buat user baru, bulan-bulan sebelum
mulai di-padding nol, dan winsorizing (§9.13) salah kira satu-satunya bulan
asli sebagai outlier ekstrem (karena kelompok pembanding nol punya varians
nol), jadi forecast diam-diam menunjukkan Rp 0 yang menyesatkan — bukan
"belum cukup data", tampak seperti benar-benar dihitung 0.

User sampaikan konteks pemakaian nyata sebagai landasan: bulan ke-2 sudah ada
pemasukan, awal bulan ke-3 baru ada 1 bulan LENGKAP data. Diimplementasikan:

1. api/reports/index.py: _earliest_posted_period() cari bulan pertama yang
   benar-benar ada transaksi POSTED — window histori forecast TIDAK PERNAH
   mundur ke sebelum bulan itu (tidak ada lagi padding nol palsu). Forecast
   juga cuma menghitung bulan yang sudah LENGKAP (bulan berjalan yang belum
   selesai dikeluarkan dari histori — membandingkan bulan penuh vs sebagian
   bikin tren keliru).
2. shared/forecast.py: holt_forecast(history, steps) — generalisasi
   linear_forecast lama jadi proyeksi N-langkah dari model tren yang SAMA
   (bukan model terpisah per horizon). 1 titik histori = proyeksi flat
   (bawa nilai sama, trend=0) alih-alih None — sesuai landasan minimal yang
   disepakati user ("1 bulan penuh" tetap dapat forecast, walau confidence
   lebih rendah dari 2+ titik).
3. Forecast dipecah 3 horizon dari model tren yang sama, masing-masing makin
   ketat syarat datanya (rule of thumb forecasting: makin jauh horizon,
   makin banyak histori yang perlu mendukungnya):
   - Jangka pendek (1 bulan depan) — butuh ≥1 bulan lengkap.
   - Jangka menengah (1 kuartal/3 bulan depan, dijumlah dari 3 proyeksi
     bulanan) — butuh ≥2 bulan lengkap (minimal ada arah tren).
   - Jangka panjang (1 tahun/12 bulan depan) — butuh ≥3 bulan lengkap.
   Tier yang belum memenuhi syarat return {"income": null, "expense": null,
   "min_real_months": N} — dashboard menampilkan "Belum cukup data — butuh
   minimal N bulan penuh riwayat transaksi", BUKAN diam-diam 0.
4. Dashboard Reports.tsx: ForecastCard dirombak jadi 3 kartu (pendek/
   menengah/panjang) + catatan jumlah bulan lengkap yang jadi dasar hitungan.

Testing: tests/test_forecast.py — linear_forecast([42]) sekarang == 42 (dulu
None; ganti kontrak sesuai landasan "1 bulan = cukup"), + holt_forecast
multi-step (flat 1 titik, ekstensi tren linear 3 titik, floor 0 per-langkah
untuk tren turun tajam). tests/test_reports.py +6: tidak ada transaksi sama
sekali, mulai bulan ini juga (0 bulan lengkap), 1/2/3 bulan lengkap membuka
